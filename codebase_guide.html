ANXI-ASSIST: Codebase, Architecture & Implementation Guide

This guide explains ANXI-ASSIST (Digital Anxiety Intervention Research Study Platform) in simple terms. It is designed to prepare you to present and defend the project's codebase, data schemas, clinical logic, and mathematics.

1. General Architecture & Technology Stack

ANXI-ASSIST is built as a **Single Page Application (SPA)** with a lightweight frontend and a cloud database backend. The system is designed to load instantly and run efficiently without heavy frameworks:

  • HTML5 & CSS3 Layout: Standard layouts configured via system classes. Theme attributes are controlled through central CSS variables for unified, fast styling.
  • Vanilla JavaScript (ES6+): Handles app routing, interactive features, rule evaluation, and statistical math entirely client-side without using heavy frameworks (like React or Angular).
  • Supabase Cloud Database: A cloud backend based on PostgreSQL that provides user authentication and secure database storage. Connection is managed directly in JavaScript via the official Supabase library.
  • Library-Free SVG Visualization: Graphs are drawn directly as vector code (SVG tags) generated by JavaScript. This eliminates heavy rendering packages, keeping code clean and responsive.

2. Database Schema & Data Flow

Relational tables map participants to their logs. The tables are configured on Supabase with the following fields and relationships:

Table Name Key Fields Relationship / Purpose
profiles id (UUID), participant_id (e.g. P-1001), is_admin 1-to-1 with Supabase Auth users. Identifies participant roles anonymously.
assessments user_id (FK), gad7 (scores), indicators (JSON) 1-to-Many with profiles. Holds weekly anxiety surveys and parameters (sleep, avoidance, tension).
journal user_id (FK), mood (1-10), note (text) 1-to-Many with profiles. Captures daily qualitative logs.
completions user_id (FK), recommendation_id (text) 1-to-Many with profiles. Tracks completed coping tasks.
coping_plans user_id (PK, FK), triggers, strategies 1-to-1 with profiles. Stores participant safety plans.
feedback user_id (PK, FK), usability scores (1-5) 1-to-1 with profiles. Stores final usability survey ratings.
Simplified Data Flow:
  1. Registration: A user signs up in [js/database.js](file:///Users/wiz/midlevel/js/database.js) using signUp(). Supabase adds them to internal auth, returns a UUID, and the code inserts a corresponding row in the profiles table with is_admin = false.
  2. Logging: When a user logs an assessment or journal entry, the code inserts a row into the database linking it to the active user's UUID.
  3. Researcher Inspection: The researcher logs in, gets redirected to `#admin`, and queries all cohort data securely.

3. Core Javascript Files & Routing

The frontend behaves like a single desktop dashboard but runs within a single page using a hash-change router:

  • [index.html](file:///Users/wiz/midlevel/index.html): The structural document. Contains all views (auth page, dashboard, progress page, admin panel) inside discrete <section class="view"> containers.
  • [js/app.js](file:///Users/wiz/midlevel/js/app.js):
    • SPA Routing (handleRouting()): Reads the URL hash (e.g. #dashboard) and toggles CSS active classes on view containers to transition screens without page refreshes.
    • Dynamic SVG Drawing (drawProgressGraph()): Computes chart coordinates and outputs raw HTML vector tags (<rect>, <path>) to render the progress trends dynamically.
  • [js/database.js](file:///Users/wiz/midlevel/js/database.js): Handles API calls to Supabase. Includes the fetchAllRows() function, which implements paginated chunking (retrieving rows in batches of 1,000) to bypass Supabase's default row retrieval limit.

4. How Clinical Coping Recommendations Are Evaluated

Instead of using statistical black-box AI, recommendations are generated through deterministic rules in [js/rules.js](file:///Users/wiz/midlevel/js/rules.js) under RulesEngine.evaluateRules(). This guarantees clinical safety and auditing transparency:

Clinical Logic Catalog:
  • Severe Anxiety Rule: If GAD-7 score $\ge 15$, recommend Crisis Guidance & Emergency Support.
  • Muscle Tension Rule: If somatic physical tension rating $> 6$, recommend Progressive Muscle Relaxation exercises.
  • Sleep Impairment Rule: If sleep quality score $< 5$, recommend Sleep Hygiene Protocols.
  • Avoidance Response Rule: If avoidance score $> 6$, recommend Systematic Avoidance Desensitization & Cognitive Reframing.
  • Impact on Functioning Rule: If GAD-7 $\ge 10$ and daily chores functioning rating $< 5$, recommend Worry Postponement & Attention Training.
  • Mild Anxiety Rule: If GAD-7 is between $5$ and $14$ (Mild/Moderate), recommend Mindful Box Breathing.
  • Worsening Symptom Trend Rule: If the latest GAD-7 score is higher than the previous score, recommend Additional Coping Plan Review.

5. How Statistical Analyses Are Calculated

The statistical analyses in the researcher dashboard ([js/statistics.js](file:///Users/wiz/midlevel/js/statistics.js)) are computed from scratch on the client-side using pure mathematical algorithms:

A. Multiple Linear Regression (Anxiety Predictors)

Goal: Determine how sleep quality ($x_1$), avoidance behavior ($x_2$), muscle tension ($x_3$), and engagement ($x_4$) predict GAD-7 anxiety scores ($Y$) using the linear model:

Y = β₀ + β₁x₁ + β₂x₂ + β₃x₃ + β₄x₄ + ε

How the Code Computes Regression:
  1. Design Matrix (X): The code builds a matrix $X$ where each row represents a check-in record, starting with a 1 (for the intercept coefficient $\beta_0$) followed by the predictor scores.
  2. Solving Matrix Normal Equations: Linear coefficients are calculated using the Ordinary Least Squares formula:
    beta = (XᵀX)⁻¹ Xᵀy where $X^T$ is the matrix transpose, and $(X^T X)^{-1}$ is the matrix inverse.
  3. Gaussian Row Elimination (Matrix Inversion): Inverting a matrix in JavaScript requires a custom solver. The function matInvert() implements Gaussian Elimination: it appends an identity matrix to $(X^T X)$ and performs row operations (swapping rows, scaling rows, adding rows) until the left side becomes the identity matrix, leaving the inverse matrix on the right.
  4. Variance Explained ($R^2$): Measures the quality of the model fit. It calculates the total sum of squares ($SS_{\text{tot}}$) and residual sum of squares ($SS_{\text{res}}$):
    R² = 1 - (SS_res / SS_tot) If $R^2 = 0.50$, it means 50% of the variations in participants' GAD-7 scores are predicted by our model.

B. Paired t-Test (Anxiety Change Significance)

Goal: Check if the difference between participants' baseline (first check-in) and final GAD-7 scores is scientifically significant, or if it is just random noise.

How the Code Computes the t-Test:
  1. Difference Score ($d_i$): For each participant, calculates the change in anxiety: $d_i = x_{i,\text{final}} - x_{i,\text{baseline}}$. A negative value indicates improvement.
  2. t-Statistic calculation: Calculates the mean difference $\bar{d}$ and standard deviation $s_d$, then computes the $t$-value:
    t = mean(d) / (std_dev(d) / sqrt(N))
  3. p-Value derivation: Calculates the two-tailed probability ($p$-value) by integrating the probability density function of Student's $t$-distribution with $N-1$ degrees of freedom. A $p < 0.05$ indicates that the score drop has a $<5\%$ likelihood of being a random occurrence, confirming statistical significance.

C. Natural Language Processing (NLP Text Trigger Analysis)

Goal: Extract common themes from the free-text description of anxiety triggers entered during assessments.

How the Code Computes NLP:
  1. Tokenization: Splits the raw text into individual words, forcing lowercase and stripping special characters.
  2. Stop Word Filtering: The code checks each word against a pre-defined blocklist (STOP_WORDS) containing standard non-informative words (e.g. the, and, of, details, log) and removes them.
  3. Frequency Sorting: Counts occurrences of the remaining words and renders a graphical chart of the top topics.
